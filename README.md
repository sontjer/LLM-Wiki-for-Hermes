# LLM Wiki for Hermes

> 基于 Obsidian 笔记库的 AI 驱动知识管理体系  
> 让大语言模型（Hermes Agent）自主完成：源文件扫描 → 自动分类 → 交叉索引 → HTML 输出  
> **源文件零修改，纯自动化，人工可控。可二开适配任意知识领域。**

---

## 这是什么

一套可复用的**知识管理自动化框架**。你只需要把原始笔记放在一个目录里，LLM Wiki 会：

1. 自动扫描并分类每一篇文档
2. 建立文档间的交叉链接
3. 生成分类索引和全目录
4. 按你的模板输出格式化 HTML

所有行为由**纯文本指令文件**控制，修改指令就等于修改 AI 的行为——无需改代码。

---

## 系统架构（三层模型）

```
源文件层（只读）
    │
    ├─→ 定时扫描 ─→ 自动分类 ─→ 复制到分类目录
    │                          ├─→ 追加 [[wikilinks]] 关联
    │                          └─→ 更新索引目录页
    │
    └─→ 用户提问 → 语义搜索 → 按指令模板 → HTML 格式输出
```

| 层级 | 位置 | 角色 | 规则 |
|------|------|------|------|
| **源文件层** | `源文件/` | 原始资料、不可修改 | 你的所有原始笔记 |
| **指令层** | `指令/` | AI 行动手册、可随时编辑 | 规则文件管控全部行为 |
| **工作区层** | `Wiki/` 分类目录 | AI 生成的加工副本、带链接和索引 | 分类目录 + 索引 + 输出 |

### 核心原则

1. **源文件不碰** — 原始笔记目录不做任何修改
2. **AI 自动化** — 分类、建链、索引更新全部由 LLM 按指令执行
3. **人工可控** — 指令文件与被管理数据同目录，打开就能看到、修改

## 组件清单

### 📂 指令文件（指令/）

这些 .md 文件决定了 AI 如何工作：

| 文件 | 作用 |
|------|------|
| `00_总纲.md` | 系统概述、工作流程、三个原则 |
| `01_分类规则.md` | 分类目录的关键词对照表、分类步骤 |
| `02_链接规范.md` | [[wikilink]] 规格、建链时机、关联规则 |
| `03_索引页模板.md` | 全目录 + 分类索引的格式规范 |
| `04_HTML生成模板.md` | HTML 输出格式、CSS 规格、保存规范 |

**你的二开起点：** 修改 `01_分类规则.md` 的关键词表即可适配任意知识领域。

### 📜 自动化脚本（脚本/）

| 脚本 | 作用 | 场景 |
|------|------|------|
| **`wiki_init.py`** | 全量初始化 + 增量同步 | 扫描 → 分类 → 复制 → 建链 → 索引 |
| **`embed_classifier.py`** | embedding 语义分类引擎 | 作为 wiki_init.py 的分类器后端 |
| **`cross_link_wiki.py`** | 跨分类反向链接增强 | 扫描全文→匹配实体→追加本页提及段 |
| **`check_links.py`** | 双向链接完整性检查 | 检出缺失链接供人工补链 |

### 🧠 Hermes Agent Skill（skill/）

- **`it-wiki-manager`** — 让 Hermes 理解整套系统运作方式的行动指南
  - 触发条件说明
  - 三层架构模式详解
  - 前置读取指令文件的步骤
  - 标准化工作流（初始化 / 增量更新 / 建链 / HTML输出）
  - 路径常量定义（按需修改）

## 快速开始

### 1. 准备源文件

把你所有笔记 .md 文件放到一个目录，例如：

```
/path/to/your-notes/
├── Cloudflare入门指南.md
├── Docker部署教程.md
├── Linux常用命令.md
└── ...
```

### 2. 修改配置

打开 `指令/01_分类规则.md`，将分类关键词改为你的知识领域分类。

打开 `wiki_init.py`，修改文件开头的路径常量：

```python
SOURCE_DIR = '/path/to/your-notes'       # 源文件目录
WIKI_DIR   = '/path/to/wiki-workspace'   # 工作区目录
```

### 3. 首次运行

```bash
python3 wiki_init.py
```

脚本会自动完成：全量扫描 → 分类 → 复制 → 建链 → 创建索引。

### 4. 接入 Hermes Agent

将 `skill/SKILL.md` 安装为 Hermes Skill，配置每日增量 cron：

```bash
hermes cron create --schedule "0 3 * * *" \
  --skill it-wiki-manager \
  --prompt "执行增量同步"
```

## 分类定制指南

### 修改分类（无需改代码）

编辑 `指令/01_分类规则.md` 的关键词表：

```markdown
| 目录名 | 收录范围 | 关键词 |
|--------|---------|--------|
| Python | Python 相关笔记 | python, flask, django, pip |
| 前端开发 | Web 前端技术 | react, vue, css, javascript |
| 数据库 | 数据库笔记 | mysql, redis, mongodb, sql |
```

然后同步更新 `wiki_init.py` 中的 `CATEGORIES` 字典即可。

### 修改 HTML 输出样式

编辑 `指令/04_HTML生成模板.md` 中的 CSS 和 HTML 骨架。

### 修改链接规则

编辑 `指令/02_链接规范.md` 调整建链策略。

## 设计哲学

1. **指令即代码** — 所有规则以 .md 文件形式存在，用户打开就能看到、修改
2. **自动化但不失控** — AI 全自动执行，但通过可编辑的指令文件保持人工终极控制权
3. **源文件保护** — 任何操作都不修改原始资料，避免误操作损坏数据
4. **零耦合** — 脚本、指令、Skill 分离，任意替换不影响其他部分

## 核心教训（来自实战）

### 链接格式双轨制

| 段落 | 格式 | 例子 | 说明 |
|------|------|------|------|
| **关联文档** | `[[xxx]]` 无前缀 | `[[古德里安]]` | Obsidian 全库搜索自动匹配 |
| **本页提及** | `[[分类/xxx]]` 有前缀 | `[[战役/斯大林格勒战役]]` | 跨分类区分同名实体 |

`[[人物/东条英机]]` 格式在 `## 关联文档` 段会失效——Obsidian 将其视为相对路径而非全库搜索。

### classify() 函数要点

1. **装备检查必须在战役之前** — 避免"轰炸机"被"轰炸"关键词截胡
2. **简繁双写** — `会战/會戰`、`抗战/抗戰`、`国民革命军/國民革命軍`
3. **短关键词防误匹配** — 用全名/长名避免歧义（如"大西"会匹配"大西洋"）
4. **人物关键词全覆盖** — 每个文件名必须有关键词命中，否则落入兜底分类

### 工作流

```
1. python3 wiki_init.py              # 编译+自动分类+关联文档
2. python3 cross_link_wiki.py        # 跨分类反向链接增强
3. python3 check_links.py            # 双向链接完整性检查
4. 手动更新 📑索引/00_全目录.md
5. 写入 log.md
```

### WebDAV 写入可靠性

davfs2 FUSE 挂载的写操作可能静默失败。重要写入（如强制补链）必须通过 **curl WebDAV PUT** 直接操作服务端。

```bash
curl -T file.md -u "user:pass" "http://webdav-server/path/file.md"
```

**HTML 输出特性**：人物跳转链接系统，点击阶段底部人物名跳转到人物群像，返回按钮回到原位置。
