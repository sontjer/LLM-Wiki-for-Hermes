# LLM Wiki 管理工作流

## 触发条件

- 用户说"更新 LLM Wiki"、"处理新文档"、"归类"、"生成索引"、"建链"、"输出 HTML"等
- 每当源文件目录有新文件加入或修改时
- 用户说"把这篇存到 LLM Wiki"等

## 架构模式：三层次模型

LLM Wiki 遵循"源文件不变 + 平行工作区 + 指令驱动"的三层架构：

| 层 | 位置 | 角色 |
|----|------|------|
| **源文件层** | `源文件/` | 原始资料，任何情况下禁止修改 |
| **指令层** | `指令/` | AI 行动手册，用户可随时编辑，AI 每次执行前重读 |
| **工作区层** | `Wiki/` 分类目录 | AI 生成的加工副本（带链接、分类归档） |

**关键原则：** 指令文件与被管理数据放在同一目录树下（而不是藏在 Agent Skill 里），用户在编辑器中直接可见可改。AI 每次执行前先读指令文件，按最新版规则操作。

## 前置读取

执行任何操作前，先读取 `指令/` 目录下所有 .md 文件，理解当前规则：
- `00_总纲.md` — 系统概述
- `01_分类规则.md` — 文档分类表
- `02_链接规范.md` — 链接格式与建链规则（重要：双轨格式）
- `03_索引页模板.md` — 索引页格式
- `04_HTML生成模板.md` — HTML 输出规范

## 工作流

### 1. 初始化（首次全量扫描）

扫描源文件目录所有 .md 文件，对每篇文档：
1. 读取文件名和前 500 字
2. 通过 embedding 语义分类（默认）或关键词匹配分配到对应目录
3. 复制到 `Wiki/分类名/` 下
4. 在副本末尾追加 `## 关联文档` 段落（使用 `[[文件名]]` 无前缀格式）
5. 更新 `📑索引/` 对应分类页

> 首次全量扫描自动构建 embedding 原型质心，缓存到 `.hermes_cache/`。后续增量在此基础上做语义匹配。
> 如无 embedding API 环境 (`SILICONFLOW_API_KEY`)，自动降级为关键词匹配。

### 2. 增量更新（新文档/修改文档）

```
python3 wiki_init.py --incremental            # embedding 模式（默认）
python3 wiki_init.py --incremental --classify-mode keyword  # 关键词模式
```

1. 遍历源文件目录，`os.stat()` 对比源文件 vs Wiki 副本的 mtime
2. embedding 语义分类或关键词匹配 → 分配目录
3. 复制并建链
4. 更新受影响分类的索引 + 全目录

> 增量模式纯文件系统操作，零网络零 LLM 消耗。无变更时 <10ms 完成。

### 3. 重建原型质心（分类规则变化时）

```bash
python3 wiki_init.py --rebuild-prototypes --classify-mode embedding
```

### 4. 建链操作（重要）

建链分两个阶段：

#### 4a. 关联文档段（同分类·无前缀）
由 `wiki_init.py` 的 `generate_cross_links()` 自动生成：
```
## 关联文档

- [[古德里安]]     ← 无分类前缀
- [[东条英机]]
```

#### 4b. 本页提及段（跨分类·有前缀）
由 `cross_link_wiki.py` 自动扫描生成：
```
## 本页提及的相关条目

- [[战役/斯大林格勒战役]]  ← 有分类前缀
- [[人物/毛泽东]]
```

#### 4c. 跨分类链接验证
运行 `check_links.py` 检测双向链接完整性：
```
python3 check_links.py
```
缺失项需人工手动补链到 `raw/` 源文件（通过 curl WebDAV PUT 写入）。

### 5. HTML 输出

1. 用户提问 → 语义搜索召回
2. 读取相关文档内容
3. 按 `04_HTML生成模板.md` 输出 HTML
4. 保存到 `Wiki/📤输出/` 目录

### 6. 完整工作流（每次变更后）

```
1. python3 wiki_init.py              # 编译 + 自动分类 + 关联文档
2. python3 cross_link_wiki.py        # 跨分类反向链接增强
3. python3 check_links.py            # 双向链接完整性检查
4. 手动更新 📑索引/00_全目录.md
5. 写入 log.md
```

## WebDAV 注意事项

davfs2 FUSE 挂载的写操作可能静默失败（文件系统显示写入成功，但实际未到达服务端）。
重要写入（如 raw/ 源文件的强制补链、目录迁移）必须使用 **curl WebDAV PUT**：

```bash
curl -T file.md -u "user:pass" "http://webdav-server/path/file.md"
```

## classify() 函数要点（关键词模式）

1. **装备检查必须在战役之前** — 避免"轰炸机"被"轰炸"关键词截胡
2. **简繁双写** — 中文关键词必须同时覆盖简体和繁体
3. **短关键词防误匹配** — 如"大西"会匹配"大西洋"，应用全名"大西泷"
4. **人物关键词全覆盖** — 每个文件名必须有关键词命中，否则落入兜底分类
5. **时间线优先检查** — 时间线类关键词应先于战役检查

## 定制指南

### 适配你自己的知识领域

| 修改项 | 位置 | 说明 |
|--------|------|------|
| 分类关键词 | `指令/01_分类规则.md` | 替换为你的领域关键词 |
| Python 脚本 | `wiki_init.py` 的 CATEGORIES | 与 01_分类规则.md 同步修改 |
| 路径常量 | `wiki_init.py` 文件开头 | 改为你的实际目录路径 |
| HTML 样式 | `指令/04_HTML生成模板.md` | 修改 CSS 适配你的品牌风格 |
| 链接规则 | `指令/02_链接规范.md` | 调整双轨格式或建链策略 |

### 二开示例

- **个人学习笔记** → 分类改为学科名（数学/物理/编程）
- **团队文档库** → 分类改为部门名（产品/设计/后端）
- **食谱管理** → 分类改为菜系（川菜/日料/烘焙）
- **论文库** → 分类改为研究方向（NLP/CV/RL）


## 自我增殖（知识缺口驱动增长）

### 适用条件
当 Wiki 已积累 ≥30 篇文档后，可用缺口探测器自动发现尚未入库的知识点。

### 调用方式
```bash
# 四支柱模式（通用，任何领域可用）
python3 脚本/gap_scanner.py /path/to/文库根

# 已知清单模式（精度更高）
python3 脚本/gap_scanner.py /path/to/文库根 --list entities.json
```

### 工作流链
```
gap_scanner 发现缺口 →
    人工确认 / opencli wikipedia 批量采集 →
    wiki_init.py 编译入库 →
    cross_link_wiki.py 跨链增强 →
    gap_scanner 二次验证
```

### 四支柱模式说明
| 支柱 | 匹配模式 |
|------|---------|
| 人物 | 2-4汉字 + 天皇/首相/将军/大将 等 |
| 事件 | 2-8汉字 + 战争/条约/运动/维新/事变 等 |
| 机构 | 2-6汉字 + 政府/军/省/部/局/会 等 |
| 概念 | 2-8汉字 + 主义/论/化/观/思想/政策 等 |

各支柱的后缀规则见 `gap_scanner.py` 顶部 PERSON_SUFFIX / EVENT_SUFFIX / ORG_SUFFIX / CONCEPT_SUFFIX 列表，可根据主题扩充。
